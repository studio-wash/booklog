## 프로젝트 아이디어

매일 독서를 **페이지 수 중심**으로 남기고, **잔디(기여도 띠)**로 읽은 양을 한눈에 보며 습관을 만든다. 감상은 **선택**으로 두어 이탈을 줄인다.

## 목적

- 독서를 꾸준히 하기 어려운 문제를 **최소 입력**으로 완화한다.
- 나중에 짧게라도 **돌아볼 수 있는 기록**을 남긴다.
- MVP에서는 **단일 색 잔디 + 창(메인 365일 또는 월 시트) 내 상대적 진하기**로 핵심 가치를 명확히 한다.

## 핵심 기능 (MVP 범위)

### 1. 메인 — 잔디

- 메인 잔디: **최근 365일**(오늘 포함) **GitHub 프로필형 띠** — 요일 **세로(Sun–Sat)**, 한 열이 **한 주**; 시간순 **왼쪽=과거·오른쪽=최근**, **가로 스크롤**. 시각 레퍼런스: `260512_github_like_ref.png`.
- **월 단위 달력**(이전·다음 월 + 해당 월 격자)은 메인 상단 **캘린더 아이콘** → 바텀시트로 제공한다.
- 날짜 칸: 기록 없음 / 기록 있음 구분. **MVP에서는 책별 색 구분 없음** (단일 팔레트). 책별 색·필터 등은 **프리미엄 후보**로 분리.
- **진하기 로직 (GitHub 스타일, 메인=최근 365일 창 기준 상대)**  
  - 메인 띠에서는 **해당 365일 안에서 가장 많이 읽은 하루 합산 페이지를 100%**로 두고, 각 날은 그날 합산 페이지에 비례해 농도 표현.  
  - 월 시트(캘린더 아이콘)에서는 **그 달 안의 최댓값**을 100%로 둔다.  
  - 데이터가 적은 초기에는 튐을 줄이기 위해 **하한 처리**를 둔다. 예: 창 최댓값이 매우 작을 때는 고정 스케일 하한을 쓰거나, 최소 1페이지 이상인 날만 단계적으로 진하게.
- 하루에 여러 권을 읽은 경우: **그날 합산 페이지 수**로 농도 계산. (MVP 시각은 단색 농도만; 칸 분할·다색은 프리미엄 검토.)
- 날짜 탭 시: 해당 일의 기록 목록(책 제목, 페이지, 선택 입력 감상 요약) 표시.

### 2. 기록 화면

- **책 선택 (필수)**  
  - 최근 읽은 책 빠른 선택.  
  - **국립중앙도서관 검색 API** 연동은 MVP에서 **보조**: 검색 → 선택 시 제목(및 가능하면 메타) 자동 채움, 실패·미검색 시 **직접 입력** 유지.  
  - 참고 링크(팀 확인용): `https://www.nl.go.kr/NL/contents/N31101030500.do` — 실제 엔드포인트·인증·이용약관은 구현 전 재확인.
- **읽은 페이지 수 (기본 노출, MVP의 주 행동)**  
  - 숫자 입력. (시작/끝 페이지 등 확장은 이후.)
- **느낀 점 (선택, 접힘)**  
  - `[ + 느낀 점 남기기 ]` 토글. **글이 제한 없음**; placeholder로 가벼운 기록을 유도. 입력 영역은 초기 1~2줄 높이, 내용에 따라 확장.
- 저장 시 메인 잔디 반영, 완료 토스트 후 메인으로 복귀.

### 3. 완독 UX

- 책에 **총 페이지 수**가 등록되어 있고, 사용자가 그날 기록으로 **완독**에 도달하면: 축하 바텀시트(또는 다이얼로그).  
- 액션: **한줄 평 남기기** / **그냥 완료** — 강제 없음.  
- 완독한 날 잔디는 동일 농도 규칙 안에서 **시각적 강조**(테두리·아이콘 등)는 구현 단계에서 태스크로 쪼갠다.

### 4. 책 관리

- 내 서재: 제목, 총 페이지(선택), 식별용 간단 메타.  
- MVP는 **로컬 우선**으로 시작해도 됨(동기·계정은 후속 플랜).

**현재 고정값**

**구현 시 준수 (나중에 이름·번들만 바꾸기 쉽게)**

1. **표시용 문자열**  
   - 스토어/설정/앱바 등 사용자 노출 문구는 코드 곳곳에 리터럴로 박지 말고, **`flutter/lib/core/app_branding.dart`**(또는 `app_config.dart`) 같은 **단일 모듈**에서만 export.  
   - 필요하면 `--dart-define=APP_DISPLAY_NAME=...` 로 빌드별 오버라이드 가능하게 설계.

2. **Android**  
   - `flutter/android/app/build.gradle(.kts)`의 `applicationId` / `namespace`만 번들의 소스 진실로 두고, 매니페스트의 `android:label`은 **`@string/app_name`** 등 리소스 한 줄만 참조.  
   - 앱 내부에서 번들 문자열이 필요하면 `package_info_plus` 등으로 **런타임 조회**; `com.studiowash.booklog`를 비즈니스 로직에 하드코딩하지 않기.

3. **iOS**  
   - `CFBundleDisplayName`(및 필요 시 `CFBundleName`)을 **xcconfig 또는 빌드 설정**으로 모으고, 코드에는 표시명 상수만 사용.

4. **딥링크·OAuth·로컬 스킴**  
   - URL scheme / redirect URI에 번들 ID를 넣을 경우 **상수 한 곳**에서만 조립해 나중에 검색·치환 가능하게.

5. **폴더 slug**  
   - `plan/PLAN-000001_haruhanjang-mvp` 등 저장소 내 폴더명은 **이력용**으로 유지해도 되며, 앱 표시명·번들 변경과 무관.

**이름/번들 변경 시 점검 순서 (체크리스트)**  
`flutter/android` Gradle `applicationId` → `flutter/ios` Xcode 타깃 `Bundle Identifier` → Android `AndroidManifest` / iOS `Info.plist` 표시명 → `flutter/lib/.../app_branding` (및 dart-define) → 스토어 콘솔 새 앱 여부(번들 변경 시 새 앱으로 올릴 수 있음) → 딥링크·푸시 등 외부 등록 URL.
