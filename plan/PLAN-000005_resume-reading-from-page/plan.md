# 피쳐 계획서

**Plan ID**: PLAN-000005  
**생성일**: 2026-05-15

## 피쳐 아이디어

책을 앱에 **처음 등록할 때** 이미 종이책·다른 앱에서 **어느 쪽까지 읽었는지**가 있을 수 있다. 지금은 읽기 기록이 없을 때 타임라인 하한이 **0쪽**으로 간주되어, 첫 기록을 예를 들어 100쪽에 두면 `pages` 델타가 100으로 잡히는 등 **앱 설치 이전 진행분**이 일일 잔디·누적 읽기량에 섞인다. 사용자가 99쪽까지 읽은 뒤 앱에 등록했다면 **다음에 잡을 쪽은 100부터**이고, 그날 읽은 분량만 델타로 세는 것이 자연스럽다.

## 목적

- 신규 등록 책에 대해 **“앱에 넣기 전까지 마지막으로 읽었던 쪽”**을 선택적으로 저장한다.
- 읽기 기록의 **절대 쪽 번호(`last_page_read`)** 규칙(FR-2)은 유지하되, **이전 진행**을 책 단위 **기준선(baseline)** 으로 두어 `insertEntry`·타임라인 검증·`pages` 재계산의 **하한**이 0이 아닌 기준선을 넘지 않도록 한다.
- 메인 카드·완독 판정·잔디 집계 등 **“이 책에서 이미 읽은 분량”**을 쓰는 경로가 모두 같은 기준선을 보도록 정리한다.

## 핵심 기능

### 1. 데이터 모델

- `books` 테이블에 **nullable 정수** 컬럼 추가 (예: `starting_last_page_read`).
  - 의미: **첫 `reading_entries` 행이 생기기 전**에 사용자가 “여기까지는 이미 읽었다”고 선언한 **절대 쪽 번호**의 상한. 엔트리가 없을 때 타임라인에서의 **가상의 마지막 쪽**과 동치.
  - `NULL` 또는 생략: 기존 동작과 동일(하한 0).
  - 값이 있을 때: `>= 0` 정수. **첫 기록의 `last_page_read`는 반드시 `> starting_last_page_read`** (같은 쪽 재등록은 “새로 읽은 진행”이 아니므로 불가). 기존 규칙 `last_page_read >= 1`과 맞추려면, 기준선이 0이면 첫 로그는 기존과 같이 ≥1, 기준선이 99이면 첫 로그는 ≥100.
- 스키마 버전 상승(예: v2→v3): `onUpgrade`에서 컬럼 추가만 수행(기존 행은 `NULL`). v2→v3 이전과 동일한 파일은 컬럼 없음 → 추가 후 기본 `NULL`.

### 2. `Book` / `insertBook` / `updateBook` / export·import

- `Book`에 필드 추가, `insertBook`·`updateBook`·`fromMap` 반영.
- **Export/import JSON**: `books` 배열 객체에 동일 키(snake_case) 포함. 포맷 버전(`export_schema_version`)이 1인 파일과의 호환: import 시 키 없으면 `NULL` 처리.
- `booklog_export_format.dart` 예시 JSON에 키 한 줄 추가(선택 필드).

### 3. 읽기 기록 로직 (`AppDatabase`)

- **`_maxLastPageReadBefore`**(또는 이를 감싼 헬퍼): 해당 책의 `starting_last_page_read`를 조회해, 엔트리 기반 쿼리 결과와 **`max(엔트리기반, 기준선)`** 으로 하한을 잡는다. (엔트리가 전혀 없으면 엔트리 쪽은 0이므로 결과는 기준선.)
- **`insertEntry`**: 위 하한을 사용해 `pages = lastPageRead - prev` 및 기존 타임라인 상·하한 검증이 일관되게 동작하는지 점검. `_reconcileReadingEntryPagesForBook`는 그대로 두되, 첫 행의 `prev`가 기준선을 반영해야 한다.
- **`lastPageBoundsForNewEntry`**: UI에 넘기는 `lowerBound`에 기준선 반영(이웃 로그만 보던 경우와 “이 책에 로그가 하나도 없음” 분기 모두).
- **`maxLastPageReadForBook`**, **`totalPagesReadForBook`**: 완독·진행률·메인 카드에서 “누적 읽은 쪽”이 **엔트리 델타 합만**이면 기준선 이전 분량이 빠진다.  
  - 권장: **표시용 최대 쪽** = `max(max(entries.last_page_read), starting_last_page_read)` (엔트리 없으면 기준선).  
  - **완독 누적 페이지**(FR-8): `totalPagesReadForBook`는 엔트리의 `pages` 합이 맞고, **총 페이지 대비 남은 분량**은 `total_pages - max(last_read_display, …)` 식으로 기준선을 포함한 “현재 쪽”을 쓴다.
- 문서화: FR-2 스펙과의 관계 — “앱 밖에서 이미 읽은 분”은 **별도 기준선 필드**로만 들어가고, 잔디·일별 델타는 **엔트리의 `pages`만** 쓰는 현재 정책을 유지하면 기준선은 **첫 로그 이후 델타가 과대가 되지 않게** 하는 역할에 집중한다.

### 4. UI

- **책 추가** 시트(검색·직접 입력 공통): 선택 입력 **“지금까지 읽은 마지막 쪽 (선택)”** — 비우면 `NULL`, 숫자면 위 규칙 검증(총 페이지가 있으면 `< total_pages` 등).
- **책 수정**: 해당 책에 `reading_entries`가 **없을 때만** 기준선 편집 허용(이미 로그가 있으면 타임라인과 충돌 방지를 위해 읽기 전용 또는 별도 정책은 구현 단계에서 최소 변경으로 확정).
- **기록 화면**: 책 선택 후 `lastPageBoundsForNewEntry`가 반환하는 하한이 기준선을 반영하므로, 첫 기록 시 기본값/힌트 문구에 “이전에 N쪽까지 읽음”이 보이면 UX가 좋다.

### 5. 테스트

- 기준선 99, 첫 로그 `last_page_read` 100 → `pages` == 1.
- 기준선 없음, 첫 로그 100 → 기존과 같이 `pages` == 100(회귀).
- 기준선 50인데 첫 로그 50 이하 시도 → 거절.
- `maxLastPageReadForBook` / 완독 경계: 엔트리 없고 기준선만 있을 때 메인/완독 UI가 기대대로인지(구현 시 스냅샷 테스트).

## 사용자 시나리오

1. 사용자가 99쪽까지 읽은 책을 검색해 추가하고, “지금까지 읽은 마지막 쪽”에 **99**를 입력한 뒤 저장한다.
2. 같은 날 기록 화면에서 해당 책을 고르고 **100**쪽까지 읽었다고 저장한다 → 일일 델타는 **1**로 잡힌다.
3. 기준선을 비운 채로만 추가한 책은 지금과 동일하게 동작한다.

## 기술적 고려사항

- 기존 사용자 DB는 컬럼 `NULL`로 마이그레이션되어 **동작 변화 없음**.
- PLAN-000004 JSON 백업: 새 컬럼을 export에 포함하면 복원 시에도 기준선이 유지된다.
- 네이밍은 구현 시 `starting_last_page_read` vs `prior_last_page_read` 중 하나로 통일(스펙·코드·JSON 동일 키).
